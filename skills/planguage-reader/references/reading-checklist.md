# Planguage reading checklist

## Read in this order

1. tag and gist
2. scale
3. meter
4. fail and goal
5. stretch and wish
6. stakeholders, owner, and source

## Translation rules

- translate the `Scale` into the real thing the team must observe
- translate the `Meter` into the evidence collection path
- translate the `Goal` into the delivery commitment
- translate the `Fail` into the red-line condition
- treat `Stretch` and `Wish` as tradeoff signals, not default commitments

## Review questions

- is the requirement really one requirement or several bundled together?
- can the scale be measured the same way twice?
- can the meter be run without hidden interpretation work?
- are conditions, user classes, workloads, or environments explicit enough?
- does any field smuggle design or solution choices into the requirement?

## Useful comparison lenses

- user stories capture stakeholder need and value, but not usually measurable limits
- acceptance criteria often correspond to goal or fail boundaries
- use cases help explain flow, while Planguage helps explain measurable success
